## 1. Bulk Import

Opening a file picker (or drag-dropping onto the Docs panel list) triggers bulk
ingestion. The user can select 30+ files at once. For each PDF:

1. Generate UUID from content hash (evid convention).
2. Auto-derive label from the PDF filename — strip extension, replace `_` and `-`
   with spaces, title-case. The user can rename in the list after the fact.
3. Run the full ingest pipeline (extract → typ → yml → vec index) in a thread pool,
   capped at `min(4, cpu_count)` concurrent workers.
4. Show a compact progress table in the panel: one row per file with status badges
   (Queued → Extracting → Indexing → Done / Error).
5. On error for a single file, mark that row red and continue with the rest. Do not
   abort the batch.

The user does not need to wait for the batch to finish before interacting with docs
that have already completed.

## 2. Anon Output Mode — Three States

The anon mode is a **set-level toggle with three positions**: Real, Placeholder, Fake.

It is surfaced as a 3-button segmented control in the sidebar (visible only for anon
sets). It resets to **Real** each time the app launches (safe default — never
accidentally export PII).

### Mode semantics

| Mode | What happens at export |
|---|---|
| Real | Text is output verbatim. No substitution. |
| Placeholder | Each entity is replaced with its `placeholder` value from the active YAML, e.g. `[PERSON A]`, `[ADDRESS 1]`, `[CPR REDACTED]`. |
| Fake | Each entity is replaced with its `fake` value from the active YAML, e.g. `Martin Andersen`, `Bredgade 88`. |

### Entity YAML schema update (two replacement columns)

```yaml
entities:
  - original: "Lars Hansen"
    variants: ["Hansen, Lars", "L. Hansen"]
    entity_type: PERSON
    placeholder: "[PERSON A]"       # auto-generated by did extract
    fake: "Martin Andersen"         # filled in by user in entity editor (or via faker)
  - original: "Roskildevej 44, 2620 Albertslund"
    variants: []
    entity_type: ADDRESS
    placeholder: "[ADDRESS 1]"
    fake: "Bredgade 88, 1260 Copenhagen"
  - original: "301284-1234"
    variants: []
    entity_type: CPR
    placeholder: "[CPR REDACTED]"
    fake: "240865-9876"
```

`did extract` auto-populates `placeholder`. The `fake` column is initially empty and
can be:
- Filled manually in the entity editor.
- Auto-generated via an optional faker pass: `AnonService.generate_fakes(yaml_path,
  language)` using the `faker` library (locale `da_DK` for Danish sets), which
  generates plausible Danish names, addresses, and CPR-format numbers.

### Implementation in AnonService

```python
def pseudonymize(self, text: str, evidence_set: EvidenceSet, mode: AnonMode) -> str:
    if mode == AnonMode.REAL:
        return text
    yaml = self.get_current_yaml(evidence_set)
    if yaml is None:
        return text   # no yaml: treat as real regardless of mode
    field = "placeholder" if mode == AnonMode.PLACEHOLDER else "fake"
    result = text
    for entity in yaml["entities"]:
        replacement = entity.get(field, "").strip()
        if not replacement:
            # fallback: if fake is empty, use placeholder
            replacement = entity.get("placeholder", "")
        for variant in [entity["original"]] + entity.get("variants", []):
            result = result.replace(variant, replacement)
    return result
```

## 3. Cross-Set Tags

### Concept

A **tag** is a named collection of document references. Tags are owned by a set
(typically a client case set) but can reference documents from any set in the
application.

The typical flow:
1. While in the general laws set, search for "psychological assessment criteria".
2. Select 3 relevant law articles from the results table.
3. Type `hansen.psych` in the tag input and click Apply.
4. The 3 law articles are now linked to tag `hansen.psych`, which lives in the
   hansen case set.
5. Later, with the hansen set active, select the tag `hansen.psych` in the Prompt
   Builder → the prompt includes both client documents tagged psych AND the 3 law
   articles tagged from the other set.

The key property: **the tag binding is the join**, not the set context. When you
export by tag, you pull from wherever those documents live.

### Tag naming convention

Tags use a two-part name: `setSlug.tagName`.

- `setSlug` is the owning set (the client case set).
- `tagName` is a short descriptive label.

When a user types a tag name without a dot (e.g. `psych`), the UI automatically
prefixes the active set's slug: `hansen-case-2024.psych`. When the current set is
the general set (not the owner), the user must type the full qualified name.

### On-disk storage

Tags are stored in a single app-level file, not per-set, because they reference
multiple sets:

```
~/.local/share/evidmgr/
  tags.yml          # global tag registry
  sets/
    ...
```

### tags.yml schema

```yaml
- name: "hansen-case-2024.psych"
  owner_set: "hansen-case-2024"
  created: "2024-11-05T10:30:00"
  items:
    - set: "hansen-case-2024"
      doc_uuid: "a3f9c1d2-..."
      chunks: null          # null = entire doc, list of ints = specific chunk indices
    - set: "eu-laws-general"
      doc_uuid: "b7e2a0f1-..."
      chunks: [2, 3]
    - set: "who-guidelines"
      doc_uuid: "c1d5f8a2-..."
      chunks: null

- name: "hansen-case-2024.liability"
  owner_set: "hansen-case-2024"
  created: "2024-11-06T09:00:00"
  items:
    - set: "hansen-case-2024"
      doc_uuid: "d4e7b0c3-..."
      chunks: null
```

## 4. Prompt Builder — Tag-Based Export

When building a prompt by tag, `TagService.resolve_items()` is called. Each resolved
item produces one evidence block in the prompt, regardless of which set it came from.

The combined result is then passed through `AnonService.pseudonymize()` with the
current anon mode. Documents from the general set (laws, guidelines) are typically
not PII-bearing and pass through unchanged; only client case documents have active
entity maps.

```
Prompt template:

You are a legal assistant. Based on the evidence below, {task}.

--- EVIDENCE ---

[1] Police report 14/2024 (case, Hansen Case 2024)
{text from client doc — anonymized per mode}

[2] Expert statement, Dr. Jensen (case, Hansen Case 2024)
{text — anonymized}

[3] Road Traffic Act § 118-122 (law, EU Laws General)
{text — passes through unchanged, no PII}

--- TASK ---
{user-defined task text}
```

Export formats:
- Plain text (above format)
- Markdown (headers per block, code fences for long extracts)
- BibTeX only (concatenates `label.bib` files from all included documents)

## 5. External Editor Integration

When the user selects a document and presses `[Open in editor]` or `[Open dir]`:

- **Open dir**: calls `QDesktopServices.openUrl(QUrl.fromLocalFile(doc.path))` to
  open the document's UUID folder in the system file manager (Finder/Nautilus/Explorer).
- **Open in editor**: calls the configured external editor with the `.typ` file path.
  The editor path is set in app preferences (default: detect VSCode → fallback to
  system default for `.typ`). On close, the app checks if `label.json` has been
  updated and, if so, regenerates `label.bib` automatically.

The VSCode snippet for typst labelling (from the evid README) should be shown in
a dismissible tip the first time a `.typ` file is opened.

## 6. Tag Creation Flow in Search Panel

The action bar in the Search panel contains:

```
[N selected]  Add to tag: [ input field ]  [Apply]     [Export prompt]
```

When the user clicks Apply with a tag name in the input:
1. If the name contains no `.`, prefix with the active set's slug.
2. If the tag does not exist yet, `TagService.create_tag()` creates it.
3. `TagService.add_items()` is called with all currently selected rows.
4. The input clears; the tag sidebar updates its count.
5. A brief confirmation toast: "3 docs added to hansen.psych".

The tag sidebar (right column of the search panel) lists all tags owned by the active
set, with their item count. Clicking a tag row in the sidebar pre-populates the tag
input and switches the results table to show only items in that tag.
